The `load.sql` data was generated from this project's Node.js scripts.

It uses the following APIs:

1. The openweathermap geo api: http://api.openweathermap.org/geo/1.0/direct
    - This is so we can get the latitude and longitude pairs from city state names.
2. The openweathermap current forecast api: https://api.openweathermap.org/data/2.5/weather
    - This is mainly to used get metadata, namely the UTC timezone offset
3. The open-meteo hourly forecasts API to get our observations: https://api.open-meteo.com/v1/forecast
    - This is so we can get our observation data and their derived alerts and weather_conditions

To run the node.js scripts:

1. Install dependencies:

```bash
npm install
```

2. Load in environment variables into `.env`:
```txt
OPEN_WEATHER_MAP_API_KEY=
# If you're using the docker compose 
DB_HOST=localhost
DB_USER=root
DB_PASS=password
DB_NAME=mydb
DATABASE_URL=mysql://root:password@localhost:3306/mydb
```

3. (Optionally) spin up mysql using `docker compose`:
```bash
npm run docker-up
```

4. Run the setup migration script:
```bash
npm run migrate-up
```

5. Run the data-loader scripts from `./scripts`:
```bash
npm run load-data
```

You can then make the load.sql file using the mysql INSERT dump:
```bash
mysqldump -h 127.0.0.1 -P 3306 -u root -p \
  --no-create-info \
  --complete-insert \
  --skip-triggers \
  --compact \
  --set-gtid-purged=OFF \
  mydb \
  states \
  cities \
  observations \
  alerts \
  > load.sql
```

We don't need to dump `weather_conditions` and `alert_types` because that's in the `setup.sql` script.

And that's it.

## Data Volume and Coverage

The dataset is generated by collecting hourly weather observations from the Open-Meteo API across multiple cities.

- **States covered:** All states included in `states` seed data
- **Time range per city:** 7-day rolling window (controlled by `INTERVAL_DAYS = 7`)
- **Granularity:** Hourly observations

This results in approximately:
- **~168 observations per city** (24 hours × 7 days)
- **Total observations:**  
  `(# cities) × 168` rows in the `observations` table
- **Alerts:** Generated conditionally; typically sparse compared to observations
- **Date range:** From the current system date at script execution, extending 7 days forward

## Deriving `weather_conditions` and `alerts` from Open-Meteo Data

The Open-Meteo API does not provide categorical weather conditions or alert objects directly. Instead, both `weather_conditions` and `alerts` are derived programmatically from raw hourly measurements.

### Weather Conditions

Each observation is assigned a condition based on deterministic rules applied to temperature and precipitation values:

- If `precipitation > 0` -> `"Rain"`
- If `temperature_2m < 0` AND `precipitation > 0` -> `"Snow"`
- If `temperature_2m > 30` -> `"Hot"`
- Otherwise -> `"Clear"`

These derived labels are mapped to entries in the `weather_conditions` table via a foreign key (`condition_id`). This allows consistent categorization and avoids storing redundant string values in the `observations` table.

---

### Alerts

Alerts are generated by evaluating threshold-based rules over the same observation data. Unlike `weather_conditions`, alerts are event-based and may span multiple timestamps.

Example rules:

- **High temperature alert:** `temperature_2m > 35`
- **Heavy precipitation alert:** `precipitation > 10`
- **High wind alert:** `windspeed_10m_kmh > threshold`

When a threshold is exceeded:
- A row is inserted into the `alerts` table
- `start_time` is set to the timestamp of the triggering observation
- `end_time` is either left `NULL` (ongoing alert) or set when the condition subsides
- `alert_type_id` references a predefined category in `alert_types`

This design separates raw measurements from derived events and allows querying both independently.

---

## Null Handling and Data Transformations

The ETL process explicitly handles missing or incomplete API data:

- Observations with missing required fields (e.g. `temperature_2m`, `time`) are skipped
- Optional fields (e.g., `pressure` or `humidity`) are inserted as `NULL` if absent
- Alerts are only generated when all required fields for evaluating a rule are present

This prevents invalid or partial rows from entering the dataset.

---

### Data Transformations

Several transformations are applied before insertion into the database:

1. **Timestamp Normalization**
   - Open-Meteo timestamps are provided in ISO format
   - They are converted to UTC using timezone offsets from OpenWeatherMap
   - Stored as `TIMESTAMP` in MySQL

2. **Unit Standardization**
   - Temperature stored in Celsius
   - Wind speed stored in km/h
   - Pressure stored in hPa
   - Ensures consistent querying and aggregation

3. **Type Conversion**
   - Floating-point values are cast to `DECIMAL`
   - Integer values (e.g., humidity) cast to `INT`
   - Prevents schema mismatch and improves precision control

4. **Row Construction**
   - Hourly arrays from Open-Meteo are iterated index-wise
   - Each index produces one observation row
   - All fields are aligned by array index

5. **Deduplication**
   - Enforced via `UNIQUE(city_id, timestamp)`
   - Duplicate inserts are ignored or skipped during loading
